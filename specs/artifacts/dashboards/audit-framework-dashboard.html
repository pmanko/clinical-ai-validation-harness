①The audit framework

The validator's job splits into two very different difficulty profiles, and the design follows that split: deterministic checks do the quantitative/temporal work soundly; a cross-family LLM does the irreducibly semantic work; a re-validate gate stops a bad flag from turning a correct answer wrong.

Deterministic floor — sound, always-on

Models are provably unreliable at quantitative/temporal claims (EQUATE: NLI ≈ majority baseline, symbolic +24.2%; Faith-and-Fate: GPT-4 ~59% on 3-digit ×). So recompute, don't entail: these checks fire only when certain, never false-positive, and feed the validator hard flags.

EQUATE 1901.03735 · Faith-and-Fate 2305.18654 · PAL 2211.10435 · ALCE 2305.14627. Lives in shared reconcile.py, used by offline judge + runtime validator.

LLM audit — decomposed, cross-family

Decompose answer + evidence into atomic claims → retrieve the chart spans → per-claim entailment (VeriFact pattern: 92.7% vs adjudicated clinicians, above clinician–clinician). A critic that conditions on the draft rubber-stamps it — CoVe's factored variant (+28% FActScore) and the cross-family rule both say keep the verifier decoupled.

VeriFact (NEJM AI 2025) · CoVe 2309.11495 · RAGAS 2309.15217 · self-preference 2410.21819

The binding risk: false flags

Self/▸under-powered critique can degrade a correct answer (Huang; Kambhampati "collapse"). Our loop is in the safe regime (external, source-grounded) only if flags are reliable. Mitigations, in priority: (1) re-validate after revising, keep-best (don't accept unconditionally); (2) gate the revision through the deterministic checks; (3) prefer span-edits over full rewrite; (4) 1 round; (5) measure flag-precision on a gold subset.

Huang 2310.01798 · Kambhampati 2402.08115 · RARR 2210.08726 · AgenticSum 2602.20040 · MEDIQA-CORR 2405.18028 (detection is the bottleneck)

Gap vs what's built today: the shipped validator (team.py::_audit_and_revise) audits holistically and accepts the revision unconditionally with a full re-synthesis. The research says add the re-validate-and-keep-best gate, the deterministic floor, and ideally per-claim decomposition. Those are the next refinements.

②Viable validators — local, cross-family, scaled to the tier

The structural win: in every tier the orchestrator is gemma and the synthesizer is qwen, so the orchestrator is cross-family vs the synthesizer by construction — and already loaded. Reusing it as the validator costs no extra model (no eviction). Viability scales with the orchestrator's size; the deterministic floor + decomposition lower the size needed, making orchestrator-reuse workable from the 12B tier up.

Why the size floor is lower than it looks

Strand-1 says small models fail holistic judgment, not necessarily decomposed per-claim entailment — and the deterministic floor removes the quantitative/temporal load entirely. So with decomposition + the deterministic helpers, a tier-matched gemma (12B/31B) is a viable grounding auditor without a cloud call.

③Audit prompts

Two focused audits with specific instructions, judged only against the chart (drafted in prompts/validation.txt). Separating context from answer localizes where the error entered — bad evidence vs bad synthesis.

CONTEXT audit — is the gathered evidence right?

  • A lab attributed to the wrong test (e.g. absolute lymphocyte count reported as CD4).
  • A value/date not in the chart, or on the wrong observation.
  • Anything the evidence asserts the chart doesn't support.

ANSWER audit — is the draft faithful & safe?

  • Trend fabricated from <2 points, or wrong direction/magnitude.
  • Window claimed beyond the data span (e.g. "past year" on 3 months).
  • Date↔value mismatch, invented date, year typo, chronology out of order.
  • Unsupported claim; failure to abstain when the chart is silent.
  • Safety: reversing a result's meaning, denying a documented allergy/adverse effect.

Verdict is schema-constrained JSON: {ok, answer_issues, context_issues} (separate fields). Spend the budget on verifier accuracy, not feedback prose — binary vs detailed feedback barely changes outcomes (Kambhampati 2310.08118). Detection, not correction, is the bottleneck (MEDIQA-CORR 2405.18028).

⑤What we've learned about tuning

A lane is not "a model" — it's model × prompt × knobs per role, plus the validator's own {model, prompt, knobs, loops}. All of it is now per-lane data, so any arm is reproducible from its level block.

Knobs move degeneration, not structural accuracy

Temperature is a degeneration↔confabulation trade-off: at temp 0 the small synths collapse into repetition loops (we saw the am-hiv-status loop and the "114 is 04-26… no, re-check" ramble), so the synth floors at 0.5 + repeat_penalty 1.15 + DRY 0.8. But knobs do not fix date math / trend-from-one-point — those are knob-insensitive. Control temp; don't make it a primary variable.

Knobs were a hidden confound — now per-lane

Knobs were global env, asymmetric per role (orchestrator DRY-off, synth DRY-on+penalty) and per path (the single-LLM arm keeps DRY), and the MLX engine silently drops DRY while llama.cpp honors it. Direct-vs-team thus confounded prompt + orchestration + sampling + serving engine. The per-role knobs: block makes each lane fully specified.

Validator design lessons (this session's research)

  • Cross-family is mandatory — self-preference doesn't shrink with capability.
  • Split the audit: grounding is decomposable (lower size floor); correctness needs the strongest reasoner.
  • False flags are the binding risk → re-validate-and-keep-best, never accept blindly.
  • Deterministic floor offloads quantitative/temporal → shrinks the LLM's load → smaller cross-family models become viable.

Validator A/B is clean even without prompt parity

Validator on vs off shares one level (same models, prompts, knobs); the audit runs deterministically and the re-synth inherits the synth knobs — only the audit round differs. So the validator's lift is isolated even while the cross-path comparison still needs harmonization.

⑥How the tuning extends to other roles & models

Per-lane knobs already cover every role

orchestrator / expert / synthesizer / validator each take per-lane temperature / repeat_penalty / dry. The same mechanism that tunes the synthesizer tunes any role — and makes each arm reproducible. The next experiment can sweep a role's knobs as a real variable instead of a global env change.

The cross-family principle generalizes

Any auditor/judge role should differ in model family from what it judges. Today gemma-audits-qwen holds for free; if prompts are harmonized to a gemma synthesizer, the validator must move off gemma — a constraint to carry forward.

The deterministic floor is shared infrastructure

reconcile.py already backs both the offline judge and the runtime validator. The Tier-1 recompute-then-diff helpers (no-trend-<2pts, window-clamp, date↔value, value-in-source, numeric-relation) are pure functions over the dated series — reusable by any role or a future second auditor. Tier-2 (RxCUI meds, allergy extraction, UCUM units) needs small envelope extensions; Tier-3 (DDI / cross-reactivity) needs a clinical KB.

Scaling & what's next

  • Validator size scales with the team class; the same pattern could scale the expert or add a second auditor.
  • Harmonize knobs + prompts across paths for a clean direct-vs-team comparison.
  • Measure validator flag-precision on a gold subset before trusting the loop.
  • This dashboard grows as runs + tuning accumulate.
