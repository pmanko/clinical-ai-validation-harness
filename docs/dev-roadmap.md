# Development Roadmap — program index

The consolidated index for the development program: north-star goals, the current state, the
**lane table** (status + gates + links into per-lane detail), the launch sequence, and the
remediation record. This doc is the single operating source of truth; each lane drills down into
its own dossier and (where it has one) a technical brief.

**How the planning surfaces fit together:**

```
specs/roadmap.canvas.tsx     VISUAL   — milestone/feature dependency model + in-flight lane cards
        │
docs/dev-roadmap.md          INDEX    — this doc: north-star, state, lane table, launch sequence, gates
        │
docs/lanes/L?-*.md           DOSSIER  — how to run a lane: branch, gates, scope, verbatim kickoff
        │
specs/artifacts/planning/*-brief.md   BRIEF — technical design / Spec Kit input (only lanes that need one)
        │
specs/0NN-*/                 SPEC     — formal requirements, generated by /speckit-specify at kickoff
```

Each layer has one job: the **dossier** carries operating facts (branch, gates, kickoff); the
**brief** carries the architecture that feeds Spec Kit; the **spec** is generated (edit the brief,
regenerate). See [`docs/lanes/README.md`](lanes/README.md).

## North-star goals

1. **Grounded validation evidence** — real systems, realistic data (the 5,284-patient corpus), every claim traceable to specific records.
2. **Single-model vs. agent-team comparison** via judged runs.
3. **Reusable harness** — the control plane (compose, manifests, provenance, validate/judge/report pipeline) generalizes to new targets; Catalyst/OpenELIS is the proof.
4. **Demo platform** — a working, stable, human- and LLM-readable showcase of local-first clinical AI with cloud parity.

## Launch sequence

L1/L2/L3 run in parallel, each in its own git worktree and session. **The worktrees were cut from an
earlier `main`** (before #27–#30) — rebase the two harness worktrees onto current `main` (or recreate)
before working in them; L1's hub worktree is independent of harness `main`.

```bash
# from the harness checkout root — recreate/refresh as needed
git -C targets/med-agent-hub worktree add ~/code/hub-wt-mcp -b feat/real-mcp-tools origin/main   # L1 (exists)
git worktree add ~/code/harness-wt-report -b feat/report-human-feedback main && (cd ~/code/harness-wt-report && uv sync --all-extras)   # L2
git worktree add ~/code/harness-wt-spine  -b docs/006-validation-spine-asbuilt main && (cd ~/code/harness-wt-spine && uv sync --all-extras)  # L3
# L5 when pulled forward:
git worktree add ~/code/harness-wt-catalyst -b 011-catalyst-fhir-sidecar-poc main
```

Harness worktrees don't need `git submodule update` (pytest ignores `targets/`), and the live stack
keeps running from the main checkout. File sets are disjoint, so lanes merge independently. The only
cross-lane edge: L1's harness pin bump waits on the hub PR merging **and** the hub-boundary smoke passing.
